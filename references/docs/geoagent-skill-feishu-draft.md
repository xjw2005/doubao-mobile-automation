# 从重复流程到 GEO Agent Skill：业务能力沉淀的一次实践

> 分享定位：本次内容聚焦 GEO Agent 与 Skill 的设计经验，重点讨论如何将重复业务流程抽象、编排并沉淀为可复用能力。
> 不展开具体业务打法、平台策略、复测规则、数据采集方式、分析模型和业务判断逻辑。

---

## 01 背景：为什么需要 Skill

在实际业务推进中，我们经常会遇到一类任务：

- 重复性强
- 流程相对固定
- 需要一定判断
- 需要记录过程
- 需要输出可复盘结果

早期这类任务可以依靠人工完成，也可以通过 RPA 做一部分自动化。但随着场景增多，单纯的“自动点击”和“自动复制”已经不够。

我们真正需要的不是让系统替人多点几下，而是把一套可复用的业务流程沉淀下来：

```text
从“人知道怎么做”
到“流程知道怎么走，Agent 知道怎么执行，人负责判断和优化”
```

这也是我们设计 GEO Agent Skill 的出发点。

---

## 02 从 RPA 到 GEO Agent + Skill

传统 RPA 很适合处理固定路径、固定页面、固定动作的重复操作。

但当任务开始涉及上下文、状态判断、异常记录和结构化输出时，仅靠动作自动化就会显得不够。

GEO Agent + Skill 的思路不是替代底层自动化能力，而是在传统自动化之上增加一层流程编排和状态决策能力。

### 对比表

| 维度 | 传统 RPA | GEO Agent + Skill |
|---|---|---|
| 关注点 | 点击、输入、复制等页面动作 | 任务目标、流程编排、结构化交付 |
| 复用对象 | 操作步骤和脚本路径 | 可复用的业务流程能力 |
| 适应性 | 强依赖固定页面和固定路径 | 通过上下文感知、状态机判断和异常分流，将固定操作流升级为可恢复、可追踪的任务流 |
| 复用性 | 依赖单机环境和脚本配置 | 以 Skill 封装任务流程，跨环境迁移时只需适配执行底座和运行参数 |
| 稳定性 | 页面变化容易导致流程中断 | 将失败节点结构化记录，支持人工复核、参数调整和持续迭代 |
| 输出方式 | 执行日志、截图或简单结果 | 结构化结果 + 异常说明 + 可追溯过程记录 |
| 人的角色 | 配置脚本、处理失败和异常补救 | 设计边界、确认关键判断、持续优化流程 |
| 价值 | 节省重复操作时间 | 沉淀业务流程资产 |

### 一句话总结

```text
RPA 复用的是动作，Skill 复用的是流程。
```

```text
我们不是让 Agent 去替代底层自动化能力，而是让 Agent 成为传统自动化之上的流程编排与状态决策层。
```

---

## 03 Human-in-the-loop 到 Agent-in-the-loop

在 Agent 开发领域，经常会提到 **Human-in-the-loop**。

它的含义是：人在关键节点介入系统决策，避免自动化完全失控。

而在这次实践里，我们其实做了一个反向补充：**Agent-in-the-loop**。

```text
Human-in-the-loop：人介入系统决策。
Agent-in-the-loop：Agent 介入传统自动化流程，承担上下文理解、状态判断、异常分流和结果结构化。
```

这并不是让 Agent 替代人，也不是让 Agent 变成一个更复杂的脚本。

更准确地说：

```text
传统自动化负责执行确定性动作；
GEO Agent 负责理解任务目标、判断流程状态、处理异常分支；
Skill 负责承载可复用的业务流程规则；
人负责边界确认、关键判断和持续优化。
```

---

## 04 Skill 设计的三个里程碑

我们在设计 Skill 时，不是从“怎么自动点击”开始，而是分成三个阶段。

### 里程碑一：从业务任务到流程模型

第一步不是写自动化，而是理解业务任务本身。

我们会先明确：

```text
输入 → 边界 → 状态 → 异常 → 输出
```

也就是：

- 用户会给什么输入
- 这个 Skill 解决什么问题
- 哪些事情不属于这个 Skill
- 执行过程中有哪些关键状态
- 哪些异常需要被标记
- 最后要输出什么结果

这一层解决的是任务建模问题：

```text
把“人知道怎么做”的经验，转成 Agent 可以执行、记录、复用的流程模型。
```

只有这个模型清楚，Skill 才不是一次性脚本，而是可以被 GEO Agent 反复调用的流程单元。

---

### 里程碑二：从固定脚本到动态编排

第二步是把流程交给 GEO Agent 执行。

这里的重点不是简单调用脚本，而是让 Agent 在执行过程中承担编排角色。

可以抽象成：

```text
上下文识别 → 状态判断 → 动作执行 → 异常分流 → 继续 / 中止
```

传统自动化更像固定路线：

```text
第 1 步 → 第 2 步 → 第 3 步 → 完成 / 失败
```

而 GEO Agent + Skill 更像任务流：

```text
识别当前状态
  ↓
选择对应流程
  ↓
执行动作
  ↓
记录结果
  ↓
遇到异常时进入分支
  ↓
继续执行或交给人工确认
```

这相当于在传统自动化之上增加了一层 Agent 编排层，使流程不再只是固定路径，而是具备状态感知和可恢复能力。

---

### 里程碑三：从执行结果到流程资产

第三步是结果沉淀。

一次执行结束后，Skill 不只是告诉我们“成功”或“失败”，而是要输出结构化结果，并保留必要的过程记录。

可以抽象成：

```text
结构化结果 → 异常记录 → 人工复核 → 流程优化 → 复用
```

这样做有两个价值：

1. 当前任务结果可以被直接查看和复盘。
2. 后续同类任务可以复用这套流程，并持续优化。

也就是说，Skill 不只是完成任务，也沉淀了团队对任务的理解方式和执行标准。

```text
从一次自动化执行，变成一份可以持续迭代的流程资产。
```

## 05 GEO Agent 在工作流中的角色

可以把 GEO Agent 和 Skill 的关系理解成：

```text
用户任务
  ↓
GEO Agent 接收上下文
  ↓
匹配对应 Skill
  ↓
Skill 执行标准流程
  ↓
生成结构化结果
  ↓
人工确认与持续优化
```

在这个过程中：

- GEO Agent 是执行底座
- Skill 是具体业务能力
- 自动化能力是底层执行工具
- 人负责边界、判断和复盘

因此，GEO Agent + Skill 的价值不是“替代所有人工”，而是让人从重复执行中释放出来，更多参与流程设计、结果判断和持续优化。

---

## 06 抽象工作流示例

以一类标准化复测任务为例，Skill 可以拆成以下环节：

```text
接收任务
  ↓
生成标准执行计划
  ↓
执行标准流程
  ↓
记录关键结果
  ↓
标记异常情况
  ↓
输出结构化结论
  ↓
人工复核
```

这里的重点不是某个平台怎么操作，也不是具体规则如何判断。

重点是：

```text
如何把一类业务流程拆成稳定、可复用、可交付的执行单元。
```

相比单纯自动化动作，Skill 更关心：

- 任务有没有被稳定执行
- 关键结果有没有被记录
- 结果是否可追溯
- 异常有没有被明确标记
- 输出能不能直接用于复盘和协作

## 07 实际落地价值

### 1. 提升执行效率

重复执行、记录和整理交给 Agent，人把精力放在判断和优化上。

```text
不是节省某一次点击，而是减少同类任务的重复执行成本。
```

### 2. 降低协作成本

以前很多流程依赖“人知道怎么做”。

现在可以逐步变成：

```text
流程知道怎么走，Agent 知道怎么执行，人负责确认和优化。
```

这对新人或跨团队协作更友好，因为大家不需要完整掌握所有背景，也能基于 Skill 完成标准化流程。

### 3. 沉淀流程资产

Skill 的长期价值，是把团队经验从个人经验沉淀为可复用能力。

```text
一次任务执行完，不只是得到一个结果，也让流程本身变得更稳定、更清晰、更可复用。
```

本次分享聚焦 GEO Agent 和 Skill 的设计经验，不展开具体业务打法。

以下内容不作为本次分享内容：

- 具体平台策略
- 复测规则
- 数据采集方式
- 分析模型
- 业务判断逻辑
- 投放 / 引流打法
- 可复现的具体操作路径

我们更关注的是：

```text
一个业务场景出现后，如何判断它是否适合 Skill 化；
如何拆解流程；
如何设计输入输出；
如何处理异常；
如何在实际工作流中落地。
```

## 09 总结

可以用三句话总结：

```text
RPA 解决的是重复操作。
GEO Agent + Skill 沉淀的是业务流程。
人的价值从执行操作，转向设计边界、确认判断和持续优化。
```

最后，可以用这句话收束：

```text
Skill 的价值，不是把某个动作自动化，而是把一套可复用的业务流程沉淀下来，让 Agent 稳定执行，让人专注判断和优化。
```

也可以用更技术化的一句：

```text
我们不是让 Agent 去替代底层自动化能力，而是让 Agent 成为传统自动化之上的流程编排与状态决策层。
```
