# GEOagent Skill 分享素材草稿

## 最终产物定位

最终目标是生成一份适合飞书云文档发布和讲解的分享汇报文档，不是 PPT。

文档应该适合边讲边看：少堆大段文字，多使用对照表、流程图、灰底提示块和简洁小节，让听众能快速理解 GEO Agent + Skill 的设计思路。

## 核心定位

这次分享不要讲具体平台策略、复测规则、数据采集方式、分析模型和业务判断逻辑。

重点讲：当一个业务场景出现后，如何判断它适不适合 Skill 化；如何拆解流程；如何设计输入输出；如何在 GEOagent 工作流里落地。

## RPA 与 GEO Agent + Skill 的区别

核心对比不是“RPA 不好”，而是：

```text
RPA 更像自动执行固定动作。
GEO Agent + Skill 更像理解任务目标后，按业务流程完成交付。
```

可以用这个主轴：

```text
RPA 复用的是动作。
Skill 复用的是流程。
```

或者：

```text
RPA 是 UI 层面的自动化。
GEO Agent + Skill 是任务层面的编排。
```

## 从“操作自动化”到“任务自动化”

可以这样讲：

```text
以前我们用 RPA，更多是在模拟人的点击、输入、复制、粘贴。
现在我们用 GEO Agent，是把一个业务任务拆成输入、执行、判断、记录、输出，让 Agent 按 Skill 完成整个任务闭环。
```

要表达的点：

- RPA 关注动作。
- Skill 关注任务。
- RPA 解决“怎么点”。
- Skill 解决“这个任务应该怎么完成”。

## 从“页面流程”到“业务流程”

可以这样讲：

```text
以前我们用 RPA，流程往往绑定在某个页面、某个按钮、某个固定路径上。
现在我们用 GEO Agent，会先理解这个任务服务什么业务目标，再把稳定流程沉淀成 Skill。
```

要表达的点：

- RPA 容易被页面变化影响。
- Skill 先定义业务边界，再执行页面操作。
- Skill 不是一段脚本，而是一套业务流程说明和执行约束。

## 从“写死脚本”到“可解释流程”

可以这样讲：

```text
以前我们用 RPA，很多逻辑藏在脚本里，出了问题只能看是哪一步点错了。
现在我们用 GEO Agent，每一步都有明确含义：为什么要执行、执行结果是什么、异常在哪里、最终结论是什么。
```

要表达的点：

- RPA 更偏执行记录。
- Skill 是过程记录 + 业务语义。
- Skill 的输出不仅说明“做完了”，还说明“结果是什么、异常是什么”。

## 从“失败就中断”到“异常可标记”

可以这样讲：

```text
以前我们用 RPA，页面元素找不到，流程可能就直接失败。
现在我们用 GEO Agent，失败不是简单中断，而是把异常类型记录下来，比如信息缺失、结果不确定、来源不可用，再交给人判断。
```

要表达的点：

- RPA 偏机械执行。
- Skill 能保留不确定性。
- 不确定时不强行给结论，而是结构化标记异常。

## 从“单次自动化”到“流程资产”

可以这样讲：

```text
以前我们用 RPA，解决的是某一次重复操作。
现在我们用 GEO Agent，是把一类业务流程沉淀成 Skill，让后续同类任务都可以复用。
```

要表达的点：

- RPA 是工具。
- Skill 是资产。
- Skill 的长期价值是沉淀团队经验，而不只是节省一次操作时间。

## 完整表达草稿

```text
以前我们用 RPA，主要解决的是“人不想重复点”的问题。比如固定页面、固定按钮、固定路径，RPA 可以很好地模拟人的操作。

但这类方式有一个限制：它更多关注的是动作本身，而不是业务目标。一旦页面变化、任务输入变化，或者中间结果需要判断，RPA 就容易变得脆弱。

现在我们用 GEO Agent + Skill，思路不一样。我们不是先问“这个按钮怎么点”，而是先问“这个任务到底要完成什么目标”。然后把业务流程拆成输入、执行、记录、异常处理和结果输出几个部分，沉淀成一个可复用的 Skill。

这样做的好处是，Agent 不只是执行动作，而是在预设边界内完成一个业务任务。它知道要保留哪些信息，遇到异常怎么标记，最后输出什么结构化结果。
```

## 飞书文档对照表：RPA vs GEO Agent + Skill

| 维度 | 传统 RPA | GEO Agent + Skill |
|---|---|---|
| 关注点 | 点击、输入、复制等页面动作 | 任务目标、流程编排、结构化交付 |
| 复用对象 | 操作步骤和脚本路径 | 可复用的业务流程能力 |
| 适应性 | 强依赖固定页面和固定路径 | 通过上下文感知、状态机判断和异常分流，将固定操作流升级为可恢复、可追踪的任务流 |
| 复用性 | 依赖单机环境和脚本配置 | 以 Skill 封装任务流程，跨环境迁移时只需适配执行底座和运行参数 |
| 稳定性 | 页面变化容易导致流程中断 | 将失败节点结构化记录，支持人工复核、参数调整和持续迭代 |
| 输出方式 | 执行日志、截图或简单结果 | 结构化结果 + 异常说明 + 可追溯过程记录 |
| 人的角色 | 配置脚本、处理失败和异常补救 | 设计边界、确认关键判断、持续优化流程 |
| 价值 | 节省重复操作时间 | 沉淀业务流程资产 |

## Human-in-the-loop 到 Agent-in-the-loop

可以这样解释：

```text
在 Agent 开发领域，大家经常提到 Human-in-the-loop，也就是人在关键节点介入系统决策，避免自动化完全失控。

而在这次实践里，我们做的是一个反向补充：Agent-in-the-loop。

它不是让 Agent 替代人，而是让 Agent 介入传统自动化流程，成为感知、判断和调度层。传统自动化负责执行确定性动作，Agent 负责理解任务目标、判断流程状态、记录异常，并把结果整理成可复用的结构化输出。
```

更适合飞书文档里的短句：

```text
Human-in-the-loop：人介入系统决策。
Agent-in-the-loop：Agent 介入传统自动化流程，承担上下文理解、状态判断、异常分流和结果结构化。
```

金句：

```text
我们不是让 Agent 去替代底层自动化能力，而是让 Agent 成为传统自动化之上的流程编排与状态决策层。
```


## 抽象案例表达

不要讲具体业务细节，可以这样抽象：

```text
比如复测类任务，RPA 可以完成“打开页面、输入问题、复制结果”这些动作。

但我们真正需要的不是复制一段文本，而是：
这个问题有没有稳定执行？
结果有没有来源？
来源是否可追溯？
异常在哪里？
最终输出能不能直接给业务同事看？

这些就不是单纯点击能解决的，而需要 Skill 设计。
```

## 金句备选

```text
如果说 RPA 是把人的手自动化，那么 GEO Agent + Skill 是把人的流程经验结构化。
```

```text
RPA 复用的是动作，Skill 复用的是流程。
```

```text
RPA 解决的是重复操作，Skill 沉淀的是业务能力。
```

```text
Skill 的本质，不是把一个动作自动化，而是把一套可复用的业务流程交给 Agent 稳定执行，并把结果变成团队可以持续使用和优化的资产。
```
